# AI Guidelines for coding agents working in this repository

These rules are **enforced** — by import-linter contracts, mypy `--strict`,
ruff, and the test suite. If you are an AI coding agent, treat a failing
check as YOUR change being wrong, never as the check being wrong.

## Architecture rules (non-negotiable)

1. **Dependencies point inward only**: `domain <- application <- adapters`,
   per bounded context (`knowledge_base`, `assistant`). The contracts in
   `pyproject.toml [tool.importlinter]` will reject violations.
2. **The domain stays pure**: no imports of fastapi, pydantic, sqlalchemy,
   langgraph, httpx, structlog, or any I/O library inside `*/domain/`.
   Domain models are frozen `@dataclass`es; value objects validate in
   `__post_init__`.
3. **Ports are `typing.Protocol`** living in `*/application/ports.py`, and
   only exist for volatile boundaries. Never import a concrete adapter in
   application or domain code.
4. **Adapters carry a technology prefix**: `SqlAlchemy*`, `PgVector*`,
   `Ollama*`, `OpenAi*`, `Pdf*`, `PlainText*`, `PydanticAi*`, `InProcess*`,
   or `LangGraph*`. The architecture tests check this.
5. **Explicit mappers at every boundary**: domain ↔ ORM (`persistence/mappers.py`),
   domain ↔ HTTP schemas (`http/mappers.py`). Do not leak framework models
   across layers.
6. **Wiring happens only in `bootstrap.py`** (the composition root). If you
   add an adapter, register it there. HTTP routers obtain use cases only via
   `Depends(bootstrap.provide_*)` and process-lived resources via `app.state`.
7. **pydantic-ai is imported in exactly ONE module**:
   `assistant/adapters/outbound/llm/pydantic_ai.py`.
8. **Bounded contexts never import each other's internals.** The only
   sanctioned bridge is `InProcessKnowledgeSearchAdapter` calling the public
   `SearchKnowledge` application use case. `shared_kernel` contains only
   concepts genuinely shared by both contexts.
9. **LangGraph and OpenTelemetry are adapters/platform concerns.** Domain and
   application must not import either.

## Code style

- Type hints on everything; `uv run --locked mypy --strict src tests` must
  pass (tests are checked too, so fakes must conform to port Protocols).
- `uv run --locked ruff check .` and
  `uv run --locked ruff format --check .` must pass.
- Docstrings explain the **why**, not the what. Write them as if teaching.
- English only: code, comments, docs, commit messages.

## Testing rules

- Unit tests use **hand-written fakes**, not mocking frameworks, and must
  never need Docker or the network.
- Anything touching PostgreSQL/pgvector goes in `tests/integration` or
  `tests/e2e` (testcontainers). Markers: `integration`, `e2e`.
- Model-quality checks belong to `evals/` and `tests/evals`, separate from
  deterministic application correctness.
- New behavior = new tests. The "no relevant context → honest refusal" path
  must always stay green.

## Verification before committing

```bash
uv run --locked ruff check . && uv run --locked ruff format --check .
uv run --locked mypy --strict src tests
uv run --locked pytest tests/unit tests/architecture # no Docker
uv run --locked pytest tests/integration tests/e2e   # needs Docker
```

## Commits

Conventional Commits: `feat:`, `fix:`, `docs:`, `test:`, `refactor:`,
`chore:`, optionally with a scope, e.g. `feat(assistant): add grading node`.
