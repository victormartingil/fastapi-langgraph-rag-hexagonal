# AI Guidelines for coding agents working in this repository

These rules are **enforced** — by import-linter contracts, mypy `--strict`,
ruff, and the test suite. If you are an AI coding agent, treat a failing
check as YOUR change being wrong, never as the check being wrong.

## Architecture rules (non-negotiable)

1. **Dependencies point inward only**: `domain <- application <- infrastructure`,
   per bounded context (`documents`, `chat`). The contracts in
   `pyproject.toml [tool.importlinter]` will reject violations.
2. **The domain stays pure**: no imports of fastapi, pydantic, sqlalchemy,
   langgraph, httpx, structlog, or any I/O library inside `*/domain/`.
   Domain models are frozen `@dataclass`es; value objects validate in
   `__post_init__`.
3. **Ports are `typing.Protocol`** living in `*/application/ports.py` (or
   `shared/application/ports.py` for cross-context capabilities). Never import
   a concrete adapter in application or domain code.
4. **Adapters carry a technology prefix**: `SqlAlchemy*`, `PgVector*`,
   `Ollama*`, `OpenAi*`, `Pdf*`, `PlainText*`, `PydanticAi*`. The architecture
   tests check this.
5. **Explicit mappers at every boundary**: domain ↔ ORM (`persistence/mappers.py`),
   domain ↔ HTTP schemas (`http/mappers.py`). Do not leak framework models
   across layers.
6. **Wiring happens only in `container.py`** (the composition root). If you
   add an adapter, register it there. HTTP routers obtain use cases ONLY via
   `Depends(container.provide_*)`.
7. **pydantic-ai is imported in exactly ONE module**:
   `chat/infrastructure/llm/pydantic_ai.py`.
8. **Bounded contexts never import each other.** Shared concepts go to the
   `shared` kernel — sparingly (it is a coupling magnet).

## Code style

- Type hints on everything; `uv run mypy --strict src tests` must pass (yes,
  tests are type-checked too — the fakes must conform to the port Protocols).
- `uv run ruff check .` and `uv run ruff format --check .` must pass.
- Docstrings explain the **why**, not the what. Write them as if teaching.
- English only: code, comments, docs, commit messages.

## Testing rules

- Unit tests use **hand-written fakes**, not mocking frameworks, and must
  never need Docker or the network.
- Anything touching PostgreSQL/pgvector goes in `tests/integration` or
  `tests/e2e` (testcontainers). Markers: `integration`, `e2e`.
- New behavior = new tests. The "no relevant context → honest refusal" path
  must always stay green.

## Verification before committing

```bash
uv run ruff check . && uv run ruff format --check .
uv run mypy --strict src tests
uv run pytest tests/unit tests/architecture          # no Docker
uv run pytest tests/integration tests/e2e            # needs Docker
```

## Commits

Conventional Commits: `feat:`, `fix:`, `docs:`, `test:`, `refactor:`,
`chore:`, optionally with a scope, e.g. `feat(chat): add grading node`.
